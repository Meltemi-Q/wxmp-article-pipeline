# Prompt + Result 案例结构

适用于 Codex、AI 工具测试、产品体验类文章。

核心不是讲“这个工具多强”，而是让读者看到：

```text
我给了什么指令
它做出了什么结果
我怎么判断这个结果
```

## 固定结构

每个案例小节只保留四块：

1. 一句场景。
2. 一条可复制 prompt。
3. 一张截图或一个结果文件。
4. 一句个人判断。

示例：

```md
## 1 先把图收好

我先试了一个很小的事：从手机发几张图，让它在电脑里收好。

> 我给你几张照片，你帮我分一下：哪些适合放文章里，哪些适合发朋友圈，哪些先别用。每张只给一句理由。

![Codex 图片整理结果](images/1-photo-sort.png)

这个点不大，但挺实用。素材先能找到，后面才谈得上写文章。
```

## 选题优先级

先看用户真实素材。用户截图里是什么用法，就优先写什么用法。

如果素材是图片和截图，优先选：

- 图片拆解
- 参考图转风格
- 文章生成学习图
- PPT / Word 整理
- 发布前图文核对

只有用户确实提供了销售/客户材料时，再写：

- 表格看重点
- 客户聊天整理

少选：

- 代码重构
- CI / PR / lint
- VPS / GitHub / token / cookie
- 公众号自动发布

## Prompt 写法

prompt 要像真人派活，不像系统说明。

好：

```text
别写复杂分析，直接告诉我三件事：哪里不对劲、明天先问谁、开会该看哪张图。
```

不好：

```text
请基于多维度业务指标构建结构化分析框架，并输出战略决策洞察。
```

## 截图要求

- 截 prompt。
- 截结果。
- 截本地文件或页面。
- 至少保留一个“不完美/待人工确认”的点。

不要只放最终漂亮结果。没有过程，读者不信。

## 图文核对

工具测试文里，请求截图和结果图要成对：

- `给我的猫参考画一个类似风格的` -> 猫咪风格图
- `根据这个文章生成一张学习用的图片` -> 学习图成品
- `把这几张截图整理成 PPT` -> PPT 预览

正文不能把 A 图说成 B 事。
